## Implementation Status

Slice 1 and the Slice 2 bounded-generation implementation shipped on `codex/ai-decision-report` on 2026-07-21. `/onboarding` sends arbitrary bounded prompts through a server-only Vercel AI Gateway seam and renders the compact, editable three-section report. The model DTO contains no trusted IDs; application code assigns IDs, verifies exact evidence excerpts, removes unsupported numeric and sensitive implementation claims, and preserves the deterministic fixture plus a safe arbitrary-prompt fallback. Timeout, refusal, malformed-output, unsupported-claim, and retry-once tests are green.

A network-enabled Gummy Alpha latency/token evaluation remains before declaring Slice 2 complete. Persistence, inline gap assistance, metric handoff, and idempotent materialization remain later slices.

## Problem Statement

Causent's current onboarding asks users to move through discrete form-like steps before they see the value of the system. The proposed experience should make the multiplicative effect of AI visible from the first prompt: a user describes what they are building, the evidence behind it, and the resources available; Causent immediately renders a partial, editable Decision Report and asks only the follow-up questions needed to make it useful.

The user remains the author and approver. AI extracts, structures, drafts, and connects work, but does not invent facts, customer evidence, metric observations, owners, costs, expected lift, or completion probability.

## Demand Evidence

This is currently a product thesis, not validated customer demand. The first goal is a tight, production-shaped design-partner test. Success must be measured through observed use, not positive reactions to a guided demo.

## Status Quo

The current Causent onboarding funnel is `paste -> editable card -> prediction commitment -> lever/watch`. It already structures pasted intent, asks mechanism questions, declares or reuses a metric, commits a decision and prediction, and transitions into lever creation. Reports are rendered from dashboard data but are seed-only on the database path. The current experience exposes several separate forms before showing a complete decision artifact.

## Target User and Narrowest Wedge

The initial user is a product or business leader starting a project who can review a decision package but does not want to manually assemble it across documents, analytics, mock-ups, and ticket systems.

The narrowest production-shaped wedge is one onboarding path:

1. Enter one prompt with optional bounded sources.
2. Receive a partially complete three-section Decision Report plus its derived assets.
3. Answer focused inline questions for missing required information while editing any field directly.
4. Approve the report progressively, even when optional fields remain visibly incomplete.
5. Confirm or add a core metric.
6. Review and approve an Action Plan of no more than seven draft actions.

## Product Principle

**The user is still in charge; AI multiplies their work.** Each meaningful user action should create several visible, inspectable outputs in text and visualization form. Multiplication must remain reversible and attributable.

One initial prompt visibly produces multiple coordinated assets:

1. The partial three-section Decision Report
2. A sourced-evidence summary with provenance states
3. A core-metric hypothesis plus an existing-metric chart when data is available
4. An Action Plan summary and one to seven editable draft actions
5. A rendered user-supplied mock-up

These are typed views of one report aggregate, not separate free-form generations. Editing the underlying field updates every dependent view.

## Decision Report Structure

### 1. Decision

- The decision being made
- Necessary background
- The problem or pain point being solved

### 2. Supporting Evidence

- Factors and supplied evidence
- Why the decision should affect the selected core metric
- Alternatives considered
- Relevant precedent from prior Causent decisions when available

### 3. Implementation

- Action Plan summary at the top
- Up to seven editable draft actions
- Estimated cost, only when supplied or explicitly confirmed
- Customers and stakeholders
- Owners
- Supplied mock-up
- Data-governance declarations
- AI-model-governance declarations

The reasoning spine is:

1. What is the decision?
2. What factors led to this decision?
3. How do we know this decision should affect the core metric?

These questions guide generation and follow-up; they are not additional top-level sections.

## Premises

1. The first prompt streams a partially completed, editable Decision Report rather than returning a prose answer.
2. Every atomic claim has a status: `sourced`, `inferred`, `suggested`, `missing`, or `user_confirmed`. A field may derive a summary status for display, but provenance belongs to claims, not the whole field.
3. The editable report remains the sole draft during onboarding. One final idempotent approval step materializes the canonical decision, metric relationship, prediction commitment, and actions so they cannot become disconnected copies.
4. Report approval requires a decision, supporting rationale, a proposed metric mechanism, and at least one proposed action. Final activation separately requires a confirmed metric relationship, the human prediction commitment, and at least one approved action. Optional Implementation fields may remain visibly missing.
5. The first-run experience presents Decision Report -> metric confirmation or Data Workshop -> Action Plan as a clear sequence. It may use sections or explicit links rather than forced conversational navigation.
6. Impact is introduced as the destination for future measurement but is not forced before evidence exists.
7. The partner wedge accepts the initial prompt, pasted supporting text, existing Causent metrics or one metric CSV, and one supplied PNG/JPEG mock-up. URL retrieval and PDF ingestion are production-ramp work.
8. User-supplied mock-ups render in the report. Original AI mock-up generation is deferred.
9. `Completion Outlook` appears only behind the design-partner feature flag as a clearly labeled concept. No unexplained numeric probability is shown and it is not part of MVP success measurement.
10. Governance begins as three optional editable declarations: data visibility (`private`, `organization`, `public`), allowed data-source notes, and approved-model notes. Automated enforcement is deferred.

## Lifecycle and Approval Semantics

The partner wedge uses one draft and one canonical write boundary:

```text
draft -> ready_for_approval -> materializing -> active
                           \-> materialization_failed -> retry
```

- `draft`: generation, inline gap questions, direct edits, metric selection, prediction commitment, and action review are stored only in the report aggregate.
- `ready_for_approval`: required report fields, a confirmed metric, the human-entered prediction, and one to seven selected actions pass deterministic validation.
- `materializing`: one idempotent server operation creates the decision, prediction, actions, and relationships.
- `materialization_failed`: the complete report remains intact; retry resumes safely using `(report_id, source_item_id)` keys.
- `active`: canonical IDs are stored on the report and the first-run experience will not repeat.

Editing an active report creates a new draft revision but never silently mutates canonical records. A later production feature may support explicit re-materialization; it is not part of the partner wedge.

## Trust and Provenance Contract

The AI may:

- Extract claims and entities from supplied material.
- Summarize evidence.
- Draft qualitative report fields.
- Suggest alternatives and actions.
- Connect supplied evidence to a proposed metric mechanism.

The AI may not silently invent:

- Customers, quotes, or research findings
- Metric observations or historical performance
- Owners or stakeholders
- Costs or dates
- Expected lift or prediction magnitude
- Completion probability

Each atomic claim stores zero or more explicit source references in Causent's schema. `suggested` content is hypothetical and renders outside the evidence presentation until a user confirms it. Anthropic strict structured output and native interleaved citations are incompatible, so provenance should be represented as Causent source-chunk IDs and excerpts rather than relying on provider-specific citation blocks.

## Guided Assistance Design

The partner wedge does not require a general chatbot or persisted conversation. Assistance is embedded into the report:

- The initial prompt generates the partial report and its derived assets.
- A deterministic gap scanner ranks required missing fields.
- The UI renders focused questions beside or beneath the relevant field.
- The model may phrase a question and propose a typed field update, but application code controls what is asked and what may change.
- Direct editing and answering a question use the same validated server commands.
- The UI explains source status, why a metric is required, and what remains before approval.
- Model failure leaves the partial report fully editable and falls back to deterministic question copy.

This preserves a production path to conversational delivery later: the report schema, gap scanner, typed commands, generation service, and provenance remain reusable. A future chat interface becomes another client of those contracts rather than a replacement.

Estimated guided-assistance work: 4-7 implementation days, overlapping report UI and generation:

- 2-3 days: typed generation, section events, validation, and deterministic fallback
- 1-2 days: gap scanner, inline questions, and typed updates
- 1-2 days: provenance guards, timeout/refusal handling, and adversarial tests

## Canonical Data Model

Add a durable report aggregate rather than storing the document only in client state:

- `decision_reports`: identity, scope, lifecycle state, active revision, onboarding progress
- `decision_report_revisions`: append-only structured snapshots or patches, author type, timestamp
- `report_sources`: pasted input/existing-record metadata and provenance
- `report_assets`: Supabase Storage references for user-supplied images and files

Revisions use append-only full JSON snapshots in the MVP. Patches are request/response transport only and contain `report_id`, `base_revision_id`, permitted field operations, and the proposed new claim values. The server rejects stale base revisions with a conflict response and returns the current revision for user reconciliation.

The structured report should have one versioned typed schema shared by model validation, server actions, persistence, and rendering. A starting contract is:

```ts
type ClaimStatus = "sourced" | "inferred" | "suggested" | "missing" | "user_confirmed";

type Claim = {
  id: string;
  text: string;
  status: ClaimStatus;
  sourceChunkIds: string[];
};

type DraftAction = {
  sourceItemId: string;
  title: string;
  summary: Claim[];
  owner: Claim | null;
};

type DecisionReportV1 = {
  schemaVersion: 1;
  decision: { decision: Claim[]; background: Claim[]; problem: Claim[] };
  supportingEvidence: {
    factors: Claim[];
    metricMechanism: Claim[];
    alternatives: Claim[];
    precedent: Claim[];
  };
  implementation: {
    actionPlanSummary: Claim[];
    actions: DraftAction[]; // 1..7 at report approval
    cost: Claim[];
    customers: Claim[];
    stakeholders: Claim[];
    assetIds: string[];
    governance: {
      dataClassification: "private" | "organization" | "public" | null;
      allowedDataSources: Claim[];
      approvedModelNotes: Claim[];
    };
  };
};
```

`dataClassification` is descriptive metadata only and never changes authorization or Storage visibility. Actual access remains scope/RLS controlled and private by default.

Runtime schema rules, not TypeScript comments, enforce cardinality and transition requirements. Missing optional content is always an empty array or `null`; `Claim { status: "missing" }` is used only for a required field that the UI must visibly request. At `ready_for_approval`, Decision, rationale, metric mechanism, confirmed metric, human prediction commitment, and `actions` must each be present; `actions` is runtime-constrained to 1-7. At `active`, the canonical decision, prediction, and at least one approved action ID are required.

Before parallel implementation begins, this contract and one complete golden JSON fixture must pass shared contract tests.

Model output never supplies trusted IDs. The server assigns immutable claim IDs and action source-item IDs when accepting a completed section. Regeneration preserves IDs only when the server deterministically matches the same logical schema path; new list entries get new IDs.

The MVP patch protocol is server-owned and versioned:

```ts
type ReportPatchV1 = {
  schemaVersion: 1;
  reportId: string;
  baseRevisionId: string;
  operations: Array<
    | { op: "replace_claim_text"; claimId: string; text: string }
    | { op: "confirm_claim"; claimId: string }
    | { op: "add_list_claim"; path: string; afterClaimId: string | null; text: string }
    | { op: "delete_list_claim"; claimId: string }
    | { op: "move_list_claim"; claimId: string; afterClaimId: string | null }
    | { op: "set_action_title"; sourceItemId: string; title: string }
    | { op: "set_action_owner"; sourceItemId: string; text: string | null }
    | { op: "add_action"; afterSourceItemId: string | null; title: string }
    | { op: "delete_action"; sourceItemId: string }
    | { op: "move_action"; sourceItemId: string; afterSourceItemId: string | null }
    | { op: "attach_asset"; assetId: string }
    | { op: "detach_asset"; assetId: string }
    | { op: "set_data_classification"; value: "private" | "organization" | "public" | null }
  >;
};
```

Clients cannot patch source IDs or set `sourced` status directly. Model proposals use the same server commands but remain `suggested` or `inferred` until provenance validation or user confirmation. The server validates paths, permissions, cardinality, asset ownership/scan status, and revision freshness, then produces a new full snapshot. A stale base revision returns `409` with the current revision ID and no partial application.

Status-dependent runtime invariants are mandatory: `sourced` requires at least one existing, authorized source chunk; `user_confirmed` records the confirming user and revision; `missing` has no text or source; `suggested` is never counted as evidence. Tombstoned sources remain resolvable to retained excerpt/hash metadata. No model or client can promote a claim to `sourced` without server validation.

Materialization is one idempotent orchestration over existing write paths. Every created row keeps its originating report, revision, and source-item ID. Exact graph:

```text
decision_report --approved revision--> decision
decision --decision_actions--> action (1..7, after action approval)
decision --human commitment--> prediction --declared metric--> metric
action --selected as watched mechanism--> lever
```

Lever creation and tracker connection remain a subsequent explicit step.

Avoid introducing a general block-editor schema in this MVP. The three report sections and their field types are prescribed and should render through purpose-built React components.

## Partner Input Security Boundary

- Text input is length-capped, treated as untrusted evidence, and can never change system instructions, tools, policies, or allowed output paths.
- The supplied mock-up is limited to one size-capped PNG/JPEG. The server verifies magic bytes, decodes it, and re-encodes it into a clean image before private Storage persistence.
- No URL fetching, PDF parsing, active-content execution, cookies, or user credentials enter the partner wedge.
- Source records have explicit retention and deletion behavior. Deleting a report revokes original-object access and schedules unshared private objects for deletion, while approved revisions retain audit-safe excerpt text, content hashes, and tombstoned source metadata. Canonical citations remain explainable without retaining the entire original file.
- Every source and chunk is tenant-scoped and covered by RLS and cross-tenant isolation tests.

## Failure and Recovery Contract

| Failure | Preserved state | User control |
|---|---|---|
| Model timeout before a section | Last complete revision and initial input | Retry missing section or edit manually |
| Malformed/invalid section | Last valid revision; rejected output logged without source content | Retry once, then deterministic empty section |
| Image decode/re-encode failure | Existing draft plus failed asset status | Replace image or continue without it |
| Stale edit conflict | Both base and current revision IDs | Review conflict and reapply selected edits |
| Materialization failure | Approved revision plus completed idempotent steps | Retry remaining steps; never recreate completed rows |
| Navigation interruption | Persisted lifecycle and guide state | Resume exact incomplete transition |

## Approaches Considered

### Approach A: Extend Existing Onboarding (Chosen)

Replace the current funnel with a persisted Decision Report workflow while retaining Causent's canonical graph and tabs. Build in milestones so the partner-review slice arrives before production hardening is finished.

Estimated for the current builder profile: 3-5 calendar weeks at 15-25 focused hours per week for the partner wedge, with the first interactive report after roughly one week and an end-to-end partner flow after roughly two to three weeks.

### Approach B: Generalized Document Workspace

Build a block editor, template engine, vector-backed source library, collaboration, arbitrary report types, and generic tools. Estimated 3-5 months for the current builder profile. Rejected because it builds a platform before validating the Decision Report.

### Approach C: Controlled Partner Demo

Build the visual flow against constrained fixtures in 3-5 days or a working but lightly hardened slice in 2-3 weeks. Rejected as the final target because successful code would require substantial rebuilding, though the first milestone intentionally resembles this slice.

This gate shows that the workflow can be used; it does not prove willingness to pay or durable demand. `Completion Outlook` remains a separate partner-review concept and does not gate the release.

### Conditional production ramp: hardening and rollout

Rollback disables new starts while preserving already-created reports for export or manual recovery.

## Parallel Agent Plan

Multiple agents can reduce implementation latency only after the shared contracts are locked.

Safe parallel lanes after Milestone 0 contract tests pass:

1. **Schema and persistence:** migrations, RLS, repositories, idempotent materialization orchestration, integration tests.
2. **Generation:** report schema, Anthropic structured output, follow-up selection, validators, adversarial tests.
3. **UI:** report renderer/editor, field states, inline questions, loading/error states.
4. **Asset path:** restricted image validation, re-encoding, Storage, and scoped reads.

Serial integration owner responsibilities:

- Own the report TypeScript/JSON schema.
- Define the materialization contract before parallel work begins.
- Merge one lane at a time.
- Run the full user journey after every merge.
- Reject generated abstractions that widen scope.

Recommended concurrency is 2-3 coding agents plus one reviewer/test agent, not an unconstrained swarm. Assign separate files or worktrees and require each lane to return tests and a written contract. Different models are useful for independent review and adversarial testing; changing models inside one implementation lane usually creates inconsistency.

## Primary Bottleneck

The largest bottleneck is not token metering. It is integrated product acceptance:

- Does the report feel immediately useful rather than like an AI-filled form?
- Are unsupported claims visibly differentiated from sourced claims?
- Does approval create the correct canonical records exactly once?
- Does the one-time guided journey survive refresh, Back, partial uploads, and model failures?
- Are generated actions specific enough to use but not falsely authoritative?

The founder's review is required at a few high-leverage checkpoints, not on every code change:

1. Approve the typed report schema and one realistic fixture.
2. Approve the interaction after watching the first end-to-end prototype.
3. Validate report logic and action quality on 5-10 adversarial scenarios.
4. Approve the production journey after a clean-account test.

Token cost should be instrumented, but it is unlikely to control schedule. Start with one provider and one production model plus observability. Introduce cheaper extraction models, caching, or routing only after measured cost or latency justifies them. Regenerate only affected fields rather than the whole report.

- Pure schema and patch-validation tests
- Golden report fixtures for representative product decisions
- Adversarial unsupported-claim tests
- Model fallback and malformed-output tests
- Idempotent approval/materialization integration tests
- RLS isolation tests for reports, sources, assets, messages, and revisions

## Success Criteria

For the first design-partner test, target:

- Median time to first editable partial report under 90 seconds.
- The user can identify which fields were sourced, inferred, missing, or confirmed.
- At least 4 of 5 structured usefulness checks pass: decision accurate, problem accurate, evidence traceable, metric mechanism plausible, next action usable.
- The user can approve the core report while optional Implementation fields remain incomplete, with a target under 15 minutes for the test scenario.
- Reaching `active` produces the intended decision, confirmed metric relationship, human prediction commitment, and one to seven editable actions with zero duplicates across forced retries.
- The guided first-run flow completes without manual recovery and never repeats.
- Zero unsupported claims are presented as sourced facts across the golden fixture and at least nine adversarial scenarios.
- The partner can explain what Causent created from their input and what still requires their judgment.
- Instrument abandonment, edit count, follow-up count, time to report approval, and time to action approval; no pass/fail threshold is claimed until the first observed partner session.
